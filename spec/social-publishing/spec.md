# Automated Social Media Publishing — Specification

## Features

### F1: Auto-Trigger on Article Publish

When an article's status changes to `PUBLISHED` (via the article processing pipeline or admin manual publish), the system automatically creates one `SocialPost` per enabled platform (Facebook, X, Instagram).

**Acceptance Criteria:**
- Article publish triggers social post creation for all connected, enabled platforms
- Each post gets a randomised delay between 5–60 minutes from article publish time
- Delays are staggered per platform (each gets an independent random delay)
- If two posts for the same platform would land within 5 minutes of each other, the later one is nudged back
- Posts are created with status `SCHEDULED` and `scheduledFor` set to the calculated time
- A BullMQ delayed job is enqueued for each post, firing at `scheduledFor`
- If no social accounts are connected, no posts are created (silent no-op)

**Trigger Point:** End of the `articleSummarise` worker (after summary + embedding complete), or when admin manually publishes a draft article.

### F2: AI Headline Generation

Each social post gets a short, platform-appropriate headline generated by Gemini, not a copy-paste of the article title.

**Acceptance Criteria:**
- Gemini generates a headline per platform, respecting character limits (280 for X incl. link+hashtags, ~200 for FB, ~150 for IG)
- Tone is informative, concise, not clickbaity — matching the configurable tone prompt
- Headline includes the article link (FB, X) or "Link in bio" CTA (IG)
- Platform owner can edit the headline during the delay window before publish
- If Gemini fails, fall back to a truncated article title

**Prompt Context:** Article title, short blurb, category, location, market, configurable tone guidelines from `SocialConfig.tonePrompt`.

### F3: Image Processing

Each post includes the article's featured image, resized for the target platform.

**Acceptance Criteria:**
- Facebook & X: landscape crop (1200x630)
- Instagram: square crop (1080x1080)
- If article has no `imageUrl`, use a branded fallback (PropertyHack logo on dark background)
- Images are processed using Sharp and saved to `/images/social/` directory
- Processed image URLs are stored per-platform in the social post record

**Implementation:** Use `sharp` (already available in Node ecosystem, no external service needed).

### F4: Hashtag Generation

Posts include auto-generated hashtags relevant to the article content.

**Acceptance Criteria:**
- System generates 2–3 hashtags for FB/X, 5–10 for Instagram
- Hashtags derived from article category, location, and market (e.g. #SydneyProperty, #PropertyNews, #AustralianRealEstate)
- Platform owner can configure default hashtags (always included) via admin settings
- Contextual hashtags are generated per article by Gemini alongside the headline
- Hashtags are appended to the post content, not stored separately

### F5: Platform API Integrations

Replace stub adapters with real API calls.

**Facebook (Graph API v19):**
- Post to Page feed: `POST /{page-id}/photos` (with image) or `POST /{page-id}/feed` (link post)
- Auth: Page Access Token (long-lived, refreshed automatically)
- Returns: post ID and permalink

**X / Twitter (API v2):**
- Upload media: `POST /2/tweets` with `media.media_ids`
- Media upload via v1.1 chunked upload endpoint (v2 doesn't support direct media)
- Auth: OAuth 1.0a (API key + access token)
- Returns: tweet ID and URL

**Instagram (Graph API via Meta):**
- Two-step: `POST /{ig-user-id}/media` (create container) → `POST /{ig-user-id}/media_publish` (publish)
- Image must be publicly accessible URL (Instagram fetches it)
- Auth: Same Meta Page Access Token as Facebook
- Returns: media ID and permalink

**Acceptance Criteria:**
- All three platforms publish successfully with image + text + link
- Per-platform results stored in `platformResults` JSON (post ID, URL, or error)
- Rate limits respected (X: 1,500 tweets/month free tier; Meta: 200 posts/hour)
- Failed posts set to `FAILED` status with error reason

### F6: Social Account Management

Admin settings page to connect and manage social platform accounts.

**Acceptance Criteria:**
- Settings page at `/admin/settings/social` shows connection status per platform
- Each platform shows: connected (green) / disconnected (grey) / expired (red)
- Connect button initiates OAuth flow (Meta) or directs to credential entry (X)
- Disconnect button stops posting to that platform without affecting others
- Auto-publish toggle per platform (on = auto-post, off = posts created as DRAFT for manual approval)
- Token refresh: Meta long-lived tokens auto-refreshed before expiry (60-day lifecycle)
- Proactive alert: if a connection is broken/expired, show a banner in admin dashboard

**Storage:** Credentials stored in `SocialAccount` model (encrypted at rest via application-level encryption of token fields).

### F7: Publishing Controls

Platform owner can manage posts during the delay window.

**Acceptance Criteria:**
- `SCHEDULED` posts can be edited (headline, image, hashtags), cancelled, or rescheduled
- Global or per-platform toggle between auto-publish and manual-approval modes
- In manual-approval mode, posts are created as `PENDING_APPROVAL` instead of `SCHEDULED`
- Pending posts require explicit "Approve" action to schedule them
- Edit/cancel available from both the post list and post detail views
- Cancelling a post removes the BullMQ delayed job

### F8: Dashboard Enhancements

Extend existing `SocialPostList` with operational features.

**Acceptance Criteria:**
- Failed posts show error reason and a "Retry" button (re-enqueues to BullMQ)
- Filter by platform (FB, X, IG, or all)
- Filter by date range
- Search by article title or post text
- Each published post shows direct link to the live post on each platform (opens in new tab)
- Quick stats bar at top: posts this week, per-platform breakdown, failed posts count

## Data Model Changes

### New Model: `SocialAccount`

```prisma
model SocialAccount {
  id            String   @id @default(cuid())
  platform      String   @unique  // 'facebook', 'twitter', 'instagram'
  accountName   String?  @map("account_name")
  accountId     String?  @map("account_id")  // platform-specific account/page ID
  accessToken   String?  @map("access_token")  // encrypted
  refreshToken  String?  @map("refresh_token")  // encrypted
  tokenExpiry   DateTime? @map("token_expiry")
  isConnected   Boolean  @default(false) @map("is_connected")
  autoPublish   Boolean  @default(true) @map("auto_publish")
  lastError     String?  @map("last_error")
  lastCheckedAt DateTime? @map("last_checked_at")
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @updatedAt @map("updated_at")

  @@map("social_accounts")
}
```

### New Model: `SocialConfig`

```prisma
model SocialConfig {
  id              String   @id @default(cuid())
  tonePrompt      String   @default("Informative, concise, neutral news tone. Not salesy or clickbaity.") @map("tone_prompt")
  defaultHashtags String[] @map("default_hashtags")  // e.g. ["#PropertyNews", "#RealEstate"]
  minPostGapMins  Int      @default(5) @map("min_post_gap_mins")
  maxDelayMins    Int      @default(60) @map("max_delay_mins")
  fallbackImageUrl String? @map("fallback_image_url")
  createdAt       DateTime @default(now()) @map("created_at")
  updatedAt       DateTime @updatedAt @map("updated_at")

  @@map("social_config")
}
```

### Modified: `SocialPost`

Add fields to existing model:

```prisma
model SocialPost {
  // ... existing fields ...
  platform        String?          // single platform per post (replaces platforms array)
  headline        String?          // AI-generated headline (before formatting)
  hashtags        String[]         // generated hashtags
  processedImage  String?  @map("processed_image")  // path to resized image
  errorReason     String?  @map("error_reason")
  retryCount      Int      @default(0) @map("retry_count")
}
```

**Migration note:** The current `platforms` array (one post → many platforms) changes to one post per platform. This is cleaner — each platform has its own headline, image dimensions, hashtags, and independent status. Existing `SocialPost` records with multiple platforms will be migrated to separate rows.

### New Enum Value: `PENDING_APPROVAL`

Add to `SocialPostStatus`:

```prisma
enum SocialPostStatus {
  DRAFT
  PENDING_APPROVAL
  SCHEDULED
  PUBLISHED
  FAILED
}
```

## Error Handling

| Scenario | Behaviour |
|----------|-----------|
| Gemini API down | Fall back to truncated article title |
| Image processing fails | Use original image URL (skip resize) |
| Platform API auth expired | Mark `SocialAccount.isConnected = false`, set `lastError`, show admin banner |
| Platform API rate limited | Retry with exponential backoff (max 3 retries) |
| Platform API generic error | Store error in `errorReason`, set post to `FAILED`, show in dashboard |
| No social accounts connected | Skip social post creation silently |
| Article has no image + no fallback configured | Post without image (FB/X support text-only; skip IG) |
